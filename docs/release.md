# Release Runbook

This is the deployment source of truth. A new session should be able to follow this file without reading old release docs.

## Current Production

- Backend URL: `https://backend.apsconsulting.kr`
- Backend image on NAS: `choho97/aps-admin-backend:1.3.12`
- App update feed: `https://update.apsconsulting.kr/win/latest.yml`
- Current app update artifact: `app/dist/APS-Admin-Setup-1.3.25.exe`
- Update server public path: `https://update.apsconsulting.kr/win`
- Local test account file, not in git: `.local/aps-test-account.md`

## Machine Roles

- Local IDE machine: code edits, app packaging, update artifact publishing. Docker CLI is not available here.
- `steve`: Docker-enabled build machine. Build and push backend images here.
- `nas`: production backend host. Do not build backend source here during normal deployment. Pull Docker Hub image and restart Compose only.
- `aligo-proxy`: fixed-IP SMS relay only. Normal app API traffic must not use this relay.

## Fixed Rules

- Do not use GitHub Actions for release.
- Do not use Git tags as the release trigger.
- Do not use `latest` for production backend deployment.
- Do not route normal app traffic through legacy `/proxy`.
- Use concrete versions such as `1.3.26`.
- `service-account.json` is mounted at runtime and must not be copied into Docker images.
- `app/.env` is development-only. Release app config is generated by the release scripts.

## Backend Release

Use this when backend code changed.

1. Pick the next backend tag.

Example:

```text
1.3.13
```

2. Build and push from `steve`.

The repo may not exist on `steve`, so copy a temporary backend build context from the local workspace:

```powershell
$version = "1.3.13"
$stage = Join-Path $env:TEMP "aps-formal-backend-$version"
Remove-Item -LiteralPath $stage -Recurse -Force -ErrorAction SilentlyContinue
New-Item -ItemType Directory -Path "$stage/backend", "$stage/scripts" | Out-Null
robocopy backend "$stage/backend" /E /XD node_modules .git /XF .env .env.* service-account.json service-account.json.backup.* npm-debug.log yarn-error.log
Copy-Item scripts/docker-build-push.sh "$stage/scripts/docker-build-push.sh"
ssh steve "rm -rf /tmp/aps-formal-backend-$version && mkdir -p /tmp/aps-formal-backend-$version"
scp -r "$stage/*" "steve:/tmp/aps-formal-backend-$version/"
ssh steve "cd /tmp/aps-formal-backend-$version && sed -i 's/\r$//' scripts/docker-build-push.sh && bash scripts/docker-build-push.sh $version && rm -rf /tmp/aps-formal-backend-$version"
Remove-Item -LiteralPath $stage -Recurse -Force
```

3. Deploy on NAS by pulling the pushed image.

```powershell
.\scripts\deploy-nas-backend.ps1 -BackendImageTag 1.3.13
```

4. Verify backend.

```powershell
Invoke-RestMethod https://backend.apsconsulting.kr/ | ConvertTo-Json -Depth 5
ssh nas "docker ps --filter name=aps-admin-backend --format 'table {{.Names}}\t{{.Image}}\t{{.Status}}'"
```

Expected:

- health response `status: ok`
- expected backend `version`
- `wsRelayEnabled: false`
- container status `healthy`

## Frontend App Release

Use this when app/Electron/frontend code changed.

```powershell
.\scripts\release-app-update.ps1 -Version 1.3.26 -SkipInstall
```

This command:

- bumps `app/package.json` and `app/package-lock.json`
- generates packaged app config for `https://backend.apsconsulting.kr`
- builds the NSIS installer
- uploads installer, blockmap, and `latest.yml` to the NAS update channel
- validates the public update feed

Verify:

```powershell
curl.exe -fsSL -H "Cache-Control: no-cache" https://update.apsconsulting.kr/win/latest.yml
```

Expected:

- `version` matches the release version
- `path` points to `APS-Admin-Setup-<version>.exe`
- public installer and blockmap checks return `200`

## Full Release Order

When both backend and app changed:

1. Build/push backend from `steve`.
2. Deploy backend to `nas`.
3. Verify `https://backend.apsconsulting.kr/`.
4. Build/publish app update locally.
5. Verify `https://update.apsconsulting.kr/win/latest.yml`.

## Translation Verification

Use after translation-related backend changes.

```powershell
$backend = "https://backend.apsconsulting.kr"
$account = Get-Content .local/aps-test-account.md -Raw
# Read credentials manually from the local-only file.
```

Login, then test one email:

```powershell
$login = Invoke-RestMethod -Method Post -Uri "$backend/auth/login" -ContentType "application/json" -Body (@{
  email = "<test email>"
  password = "<test password>"
} | ConvertTo-Json)
$headers = @{ Authorization = "Bearer $($login.accessToken)" }
Invoke-RestMethod -Method Post -Uri "$backend/email-inquiries/657/translate" -Headers $headers -ContentType "application/json" -Body "{}"
```

Backfill or force-regenerate existing translations:

```powershell
Invoke-RestMethod -Method Post -Uri "$backend/email-inquiries/translations/backfill" -Headers $headers -ContentType "application/json" -Body (@{
  limit = 100
  force = $true
} | ConvertTo-Json)
```

DB summary:

```powershell
ssh nas "docker exec aps-postgres psql -U apsuser -d aps_admin -c `"SELECT COALESCE(translation_status,'<null>') AS status, COALESCE(detected_language,'<null>') AS lang, COUNT(*) FROM email_inquiries WHERE is_outgoing=false GROUP BY 1,2 ORDER BY 1,2;`""
```

Expected current summary:

- English/non-Korean incoming mail: `completed`
- Korean mail: `not_required`
- failed count: `0`

## Common Failure Notes

- If app update does not appear, compare installed app version with remote `latest.yml`.
- If backend route is 404 after code changes, the NAS is probably still running an older backend image.
- If translation returns short output, confirm backend is using full ZOHO content, not stored preview text.
- If Docker build/push is needed, do it on `steve`, not the local IDE machine.
- If deployment script fails with `$'echo\r'`, ensure remote script stdin strips CR characters before `bash`.
