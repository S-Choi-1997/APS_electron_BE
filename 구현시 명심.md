목표:
이 작업 마일스톤을 실제 운영/사용 흐름에서 믿고 쓸 수 있게 완료해라.
테스트를 통과시키는 것이 목표가 아니다. 테스트는 완료를 입증하는 증거 중 하나다.

기준 문서/요구사항:
<요구사항 문서/이슈/목표>

완료의 의미:
- 실제 사용자가 겪는 주요 흐름에서 동작해야 한다.
- 호출자/사용자가 실수하기 쉬운 입력은 명확히 차단하거나 복구해야 한다.
- 실패 시 원인을 이해할 수 있는 오류, 로그, artifact, 응답이 남아야 한다.
- 기존 안전장치, 권한 경계, 데이터 보호 정책을 약화하면 안 된다.
- 코드, 테스트, 문서, API/CLI/UI 출력이 서로 같은 동작을 설명해야 한다.
- 테스트만 통과하고 실제 사용 경로가 깨져 있으면 완료가 아니다.

금지:
- 테스트만 통과하도록 hardcode하기
- 특정 fixture, 특정 id, 특정 현재 상태에만 맞춘 처리
- 실패를 숨기거나 status만 바꾸기
- 안전장치를 느슨하게 풀어서 성공률 올리기
- 요구사항 밖 기능을 섞어서 완료 범위 흐리기
- “나중에”, “대체로”, “일단” 같은 말로 미완료 항목 닫기

작업 절차:

1. 먼저 Product Intent를 재정의해라.
   - 이 변경이 누구의 어떤 문제를 해결하는가
   - 실제 사용 흐름에서 성공은 어떤 모습인가
   - 실패하면 어떤 운영 문제가 생기는가
   - 절대 깨면 안 되는 기존 경계/안전장치는 무엇인가

2. 요구사항을 체크리스트로 뽑아라.
   각 항목에 실제 사용 시나리오를 붙여라.

3. 구현 전 Acceptance Matrix를 만들어라.
   각 행은 다음을 포함해야 한다:
   - Requirement
   - Real usage scenario
   - Code path
   - Expected output / error / artifact
   - Verification method
   - Risk if faked or skipped

4. 구현해라.
   범위는 요구사항을 닫는 데 필요한 만큼만 넓혀라.
   관련 결함이 완료를 막으면 고치되, unrelated refactor는 하지 마라.

5. 검증해라.
   프로젝트의 기본 검증 명령을 실행해라.
   추가로 이 변경의 실제 사용 경로를 직접 검증하는 테스트나 명령을 실행해라.

6. 테스트 외 검증을 해라.
   실제 호출자/사용자가 받는 출력, 오류, 로그, artifact, UI/API 응답을 확인해라.
   테스트가 내부 구현만 확인하고 실제 사용 결과를 보지 못하면 검증이 부족한 것이다.

7. 최종 전에 스스로 편향을 지우고, 비관적 Production Review를 해라.
   아래 질문에 답해라:
   - 실제 사용 경로가 동작하는가?
   - 실패 경로가 명확하고 복구 가능하거나 안전하게 차단되는가?
   - 기존 안전장치와 호환성이 유지되는가?
   - 오래된 데이터/설정/호출자와 호환되는가?
   - 문서와 실제 구현이 일치하는가?
   - 이 변경을 배포해도 되는가?

8. 최종 답변에는 반드시 포함해라:
   - Product Intent 충족 여부
   - Acceptance Matrix 결과 요약
   - 변경 파일
   - 검증 명령 결과
   - 남은 blocker 또는 residual risk
   - 완료라고 부를 수 없는 항목은 명확히 partial/blocked로 표시

완료 선언 조건:
모든 핵심 요구사항이 code evidence + runtime/output evidence + verification evidence를 가져야 한다.
증거 없는 항목은 완료가 아니다.
